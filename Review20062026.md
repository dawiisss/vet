## CRITICAL — Security / Data Leaks / Crashes

### 1. SSH passwords sent to renderer in plaintext over IPC
**File:** `src/main/config.ts:308-329`

`loadConfig()` decrypts SSH passwords/passphrases into `currentConfig`. Then:
- `config:changed` IPC event emits the full `currentConfig` (decrypted passwords included)
- `config:get` returns `currentConfig` with decrypted passwords
- `config:set` emits `config:changed` with decrypted passwords
- `sanitizeConfig` does **not** strip passwords

**Fix:** Strip sensitive fields (passwords, passphrases, private keys) from the config before sending to the renderer.

---

### 2. Path traversal — arbitrary filesystem reads from renderer
**File:** `src/main/workspace.ts:32-120`

`listDir()` and `readFileHead()` accept arbitrary directory/file paths from the renderer with **zero validation**. A compromised renderer could:
- Call `workspace:list-dir` with `/etc/` to enumerate system files
- Call `workspace:read-file-head` with `/etc/shadow` or `~/.ssh/id_rsa`

**Fix:** Validate paths are within allowed directories (home, project), or maintain an allowlist.

---

### 3. Cross-window terminal injection
**File:** `src/main/ipc/terminalHandlers.ts:150-155`

The `terminal:write` handler has **no check that the terminal `id` belongs to the calling window**. A renderer from any window can write arbitrary data (including escape sequences) to any terminal in any other window.

```ts
ipcMain.on("terminal:write", (_event, { id, data }) => {
  writeToTerminal(id, data); // No window ownership check
});
```

**Fix:** Compare against `windowTerminals` map (same pattern used in `terminal:destroy` handler at lines 83-89).

---

### 4. Event listener leak in adblocker
**File:** `src/main/adblocker.ts:252-271`

`blocker.on("request-blocked")` listener is added on every `enableAdblocker()` call but **never removed**. `disableAdblocker()` only calls `blocker.disableBlockingInSession()` without removing the listener. Toggling adblock on/off accumulates listeners, causing:
- Memory leak
- Blocked count multiplied incorrectly in the UI

**Fix:** Store the listener reference and call `blocker.removeListener()` in `disableAdblocker`.

---

### 5. SQLite database never closed
**File:** `src/main/historyDb.ts:453-466`

`cleanupHistoryDb()` properly closes the SQLite connection and clears the flush interval, but **it is never called anywhere** in the codebase. Consequences:
- Database connection never closed — risk of WAL/journal corruption on exit
- `flushInterval` runs every 500ms forever
- `writeBuffer` array remains in memory

**Fix:** Call `cleanupHistoryDb()` in `app.on("before-quit")` or `app.on("will-quit")`.

---

### 6. `eval` on `/etc/os-release` in installer
**File:** `install.sh:53`

```sh
eval "$(grep -E '^(ID|ID_LIKE)=' /etc/os-release)"
```

Using `eval` on file content is a code injection risk. Use `source` (`.`) or parse manually.

**Fix:** Replace with `. /etc/os-release` or manual parsing with `sed`/`awk`.

---

### 7. `ts-jest@29` paired with `jest@30`
**File:** `package.json:107,111`

`ts-jest` v29 targets Jest v29. The project uses Jest v30, but `ts-jest@30` does not exist on npm. This major version mismatch can cause test failures, broken TypeScript compilation, or silent test-skipping.

**Fix:** Either downgrade Jest to v29 or verify `ts-jest@30` availability and upgrade.

## HIGH — Resource Leaks / Racing / Performance

### 8. Sysinfo polling loop never stops
**File:** `src/main/sysinfo.ts:5-28`

The `scheduleNext` chain calls `setTimeout` every 2 seconds. Once started via `sysinfo:start`, it runs forever unless `sysinfo:stop` is explicitly called — even after the main window is closed. The `isDestroyed()` check only skips sending; the timeout chain continues.

**Fix:** Add cleanup on `app.on("before-quit")` or `mainWindow.on("closed")` to stop the loop.

---

### 9. Orphaned intervals in updater simulation
**File:** `src/main/ipc/updaterHandlers.ts:92-133`

In the `updater:simulate` and simulated `updater:download` handlers:
- `setInterval(() => {...}, 150)` runs with no guard against double-start
- If user closes the dialog or quits the app, the interval runs forever
- `isSimulationActive` flag doesn't clear existing intervals

**Fix:** Store and clear the interval on dialog close, app quit, and before starting a new one.

---

### 10. Race condition in SFTP session
**File:** `src/main/sftp.ts:108-145`

`getSftpSession` registers three mutually-conflicting handlers on `client`:
- `ready` → sets up SFTP and resolves
- `error` → rejects
- `close` → deletes session from `sftpSessions` map

If `close` fires before the `ready` handler's `client.sftp()` callback completes, the session is removed from the map before it was ever stored. Subsequent calls will try to reconnect, but intermediate state is lost.

**Fix:** Only remove from `sftpSessions` in the `close` handler if the session was already successfully established.

---

### 11. Chokidar FSWatcher never closed
**File:** `src/main/config.ts:295-301`

The chokidar config file watcher is stored in `let watcher: FSWatcher | null = null` (line 117) but **never closed** — no `watcher.close()` call anywhere. This is a resource leak on app shutdown.

**Fix:** Call `watcher.close()` in `app.on("before-quit")`.

---

### 12. `paletteActions` array rebuilt every render
**File:** `src/renderer/src/shared/components/ModalManager.tsx:63-178`

A 25+ item array of `CommandAction` objects with inline `onExecute` handlers is reconstructed on every render of `ModalManager`. Passed through to `CommandPalette` → `useSearchableList`, triggering re-filtering and re-rendering of the entire command palette on every parent render.

**Fix:** Wrap in `useMemo` with proper dependency array.

---

### 13. Terminal config effect depends on entire `config` object
**File:** `src/renderer/src/features/terminal/hooks/useTerminal.ts:427-466`

```ts
useEffect(() => {
  // updates fontFamily, fontSize, cursorBlink, cursorStyle,
  // theme, scrollback, webglEnabled, etc.
}, [config, terminal, terminalId]);
```

Any config change (even `sidebarOpen` or `keybindings`) causes this effect to fire on **every active terminal**, re-applying all xterm.js options and potentially re-creating the WebGL addon.

**Fix:** Destructure only the specific config properties used in the dependency array.

### 14. Shell whitelist bypassed
**File:** `src/main/pty.ts:56-67`

```ts
// Lines 62-67 bypass the SAFE_COMMANDS whitelist:
if (existsSync(shellPath)) {
  if (stat.isFile()) { return true; }
```

The whitelist is effectively useless — any executable on the system passes validation as long as it exists on disk. Only nonexistent paths are blocked.

**Fix:** Remove the exists-check bypass or rework the validation to actually enforce the whitelist.

### 29. Dead `allowScripts` field in package.json
**File:** `package.json:123-125`

### 30. Blocking `VACUUM` and while-loop in `pruneHistory`
**File:** `src/main/historyDb.ts:335-356`

The size-based pruning loop runs synchronous `fs.statSync` and `db.exec("DELETE ...")` in a while-loop. `VACUUM` (also at lines 285 and 448) rewrites the entire database file synchronously, **freezing the main process** for potentially seconds on large databases.

**Fix:** Use `db.pragma("auto_vacuum = INCREMENTAL")` and incremental vacuum. Consider moving pruning to a worker thread or running it during idle periods.
